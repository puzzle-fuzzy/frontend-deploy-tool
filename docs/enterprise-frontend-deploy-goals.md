# 企业级前端部署工具｜目标文档

## 1. 产品定位

本产品定位为 **中小公司内部使用的前端项目发布中心**。

它不是一个简单的 `dist` 文件上传工具，也不是一开始就对标 Vercel / Netlify 的完整云平台，而是为企业内部多个前端项目提供统一的：

- 项目管理
- 构建产物上传
- 版本管理
- 预览访问
- 正式发布
- 一键回滚
- 权限控制
- 操作审计
- 静态资源访问

核心目标是让公司内部的前端项目发布流程从“手动上传服务器”升级为“可追踪、可回滚、可控制的发布系统”。

---

## 2. 要解决的问题

当前中小公司常见的前端部署问题包括：

1. 前端项目分散，访问地址不统一。
2. 测试环境、预发环境、正式环境混乱。
3. 发布依赖人工复制文件，容易出错。
4. 上传后直接覆盖线上版本，无法安全预览。
5. 上线后出现问题，无法快速回滚。
6. 谁发布了什么、什么时候发布的、发布到哪里，很难追踪。
7. 权限不清晰，任何人都可能覆盖正式环境。
8. React / Vue 等 SPA 项目刷新深层路由容易 404。
9. 静态资源缓存策略不规范，导致用户访问到旧版本。
10. 不同项目的部署规则、域名、版本历史缺乏统一管理。

## 3. 目标用户

### 3.1 主要用户

| 用户角色 | 诉求 |
|---|---|
| 前端开发 | 快速上传构建产物，生成预览地址，发布新版本 |
| 测试人员 | 访问指定预览版本，验证功能是否正常 |
| 产品经理 | 查看项目当前版本、预览新功能、确认发布 |
| 技术负责人 | 控制正式发布、回滚版本、查看操作记录 |
| 运维人员 | 管理域名、访问规则、部署目录、缓存策略 |

### 3.2 第一阶段重点服务对象

第一阶段主要服务：

- 前端开发
- 技术负责人
- 测试人员

优先解决“项目发布闭环”，而不是做复杂的企业组织管理。

---

## 4. 产品目标

### 4.1 核心目标

建立一个内部前端部署平台，让每个前端项目都可以完成以下闭环：

```text
创建项目
↓
上传构建产物
↓
生成预览版本
↓
测试确认
↓
发布到正式环境
↓
出现问题一键回滚
↓
保留版本记录和操作日志
```

### 4.2 体验目标

产品应该给用户一种明确感：

```text
公司所有前端项目都在这里
每个项目当前线上版本是什么，可以一眼看到
上传新版本后不会直接影响线上
发布前可以预览
出问题可以快速回滚
谁操作过什么都能查到
```

### 4.3 管理目标

- 项目统一管理
- 版本不可变
- 正式发布可控
- 历史版本可追踪
- 操作行为可审计
- 权限边界清晰

---

## 5. 非目标范围

第一阶段不追求做成完整的云原生部署平台。

以下能力不作为第一阶段目标：

- Kubernetes 部署
- 完整 CI/CD 编排系统
- 在线自动拉取 Git 仓库构建
- 复杂审批流
- 多租户计费
- 边缘函数
- Serverless Functions
- A/B 测试
- 灰度发布
- 复杂监控大盘
- 完整 CDN 管理平台

第一阶段只聚焦一个问题：

> 把前端构建产物安全、清晰、可回滚地发布出去。

---

## 6. 核心产品原则

### 6.1 上传不等于上线

上传构建产物后，只生成一个预览版本。

正确流程：

```text
上传产物 → 生成 Preview → 测试通过 → 发布为 Production
```

不能上传后直接覆盖正式环境。

---

### 6.2 版本不可变

每次上传都生成一个独立版本。

版本一旦生成，不应该被修改，只能：

- 发布
- 预览
- 回滚引用
- 归档
- 删除

这样才能保证回滚可靠。

---

### 6.3 发布是切换指针，不是覆盖文件

底层应采用版本目录结构。

示例：

```text
/projects/tsing-tao/versions/v1
/projects/tsing-tao/versions/v2
/projects/tsing-tao/versions/v3
/projects/tsing-tao/current -> v3
```

发布正式版时，本质上是把 `production` 环境从旧版本指向新版本。

回滚时，则把 `production` 环境重新指回历史版本。

---

### 6.4 正式环境必须可回滚

任何正式发布都必须保留历史版本。

上线后出现问题时，用户应该可以：

```text
选择历史版本 → 点击回滚 → 正式环境恢复旧版本
```

---

### 6.5 企业级系统必须有审计日志

所有关键操作必须记录：

- 谁创建了项目
- 谁上传了版本
- 谁发布了正式环境
- 谁执行了回滚
- 谁删除了版本
- 谁修改了项目设置
- 谁修改了访问规则

---

## 7. 核心能力范围

### 7.1 项目管理

每个前端项目应该有独立空间。

项目信息包括：

| 字段 | 说明 |
|---|---|
| 项目名称 | 给人看的名称 |
| 项目 Slug | 用于访问路径 |
| 项目描述 | 项目用途说明 |
| 项目类型 | React / Vue / Astro / Static / H5 |
| 创建人 | 谁创建的 |
| 当前正式版本 | 当前线上运行版本 |
| 正式访问地址 | 线上访问 URL |
| 最近发布时间 | 最后一次发布记录 |
| 版本数量 | 历史部署数量 |

---

### 7.2 环境管理

第一阶段建议保留两个环境：

| 环境 | 说明 |
|---|---|
| Preview | 上传后自动生成的预览版本 |
| Production | 正式线上版本 |

后续可以扩展为：

```text
Development
Test
Staging
Production
```

---

### 7.3 版本管理

每次上传生成一个 Deployment。

Deployment 应包含：

| 字段 | 说明 |
|---|---|
| 版本 ID | 系统生成的唯一 ID |
| 所属项目 | 归属哪个项目 |
| 状态 | Preview / Production / Archived / Deleted / Failed |
| 文件大小 | 当前版本总体积 |
| 文件数量 | 当前版本文件数 |
| 上传人 | 谁上传的 |
| 上传时间 | 什么时候上传 |
| 发布人 | 谁发布到正式环境 |
| 发布时间 | 正式发布时间 |
| 版本备注 | 本次部署说明 |
| Commit 信息 | 后续接入 CI 后使用 |

---

### 7.4 上传与产物校验

第一阶段支持：

- 上传 `dist.zip`
- 上传静态文件夹
- API 上传
- 后续支持 CI 上传

上传后必须校验：

| 校验项 | 说明 |
|---|---|
| 是否存在 `index.html` | 前端入口文件 |
| 文件总大小 | 防止超大产物 |
| 文件数量 | 防止异常上传 |
| 是否包含危险文件 | 如 `.env`、`.git`、私钥文件 |
| 是否包含 `node_modules` | 防止误传源码依赖 |
| 是否需要 SPA fallback | React / Vue history 路由需要 |
| 静态资源路径是否完整 | 避免 assets 丢失 |

需要拦截的高风险内容：

```text
.env
.env.local
.pem
.key
id_rsa
.git/
node_modules/
.DS_Store
```

---

### 7.5 发布与回滚

版本操作应包括：

| 操作 | 说明 |
|---|---|
| 预览 | 打开该版本的独立预览地址 |
| 发布为正式版 | 将该版本切换为 Production |
| 回滚到此版本 | 将 Production 指回历史版本 |
| 删除版本 | 删除无用版本 |
| 查看文件列表 | 查看上传产物内容 |
| 查看部署日志 | 查看上传、校验、发布过程 |
| 添加备注 | 标记版本用途 |

---

### 7.6 访问地址设计

第一阶段可使用统一域名路径访问：

```text
https://super.yxswy.com/deploy/{projectSlug}
```

版本预览地址：

```text
https://super.yxswy.com/deploy/{projectSlug}/preview/{deploymentId}
```

后续支持自定义域名：

```text
https://admin.company.com
https://activity.company.com
https://www.company.com
```

---

### 7.7 SPA 路由支持

项目设置中必须支持路由模式：

```text
静态站点
单页应用 SPA
```

当项目为 SPA 模式时，所有未命中的路由应回退到 `index.html`。

这对 React Router / Vue Router 项目非常重要。

---

### 7.8 缓存策略

默认缓存策略：

```text
index.html:
Cache-Control: no-cache

assets/*.js
assets/*.css
assets/*.png
Cache-Control: public, max-age=31536000, immutable
```

原则：

- `index.html` 不强缓存，保证用户可以拿到最新入口。
- 带 hash 的静态资源可以强缓存，提高访问速度。

---

### 7.9 权限系统

第一阶段建议三种角色：

| 角色 | 权限 |
|---|---|
| 管理员 | 创建项目、管理成员、发布正式版、回滚、删除版本 |
| 开发者 | 上传版本、生成预览、查看部署记录 |
| 只读成员 | 查看项目、访问预览、查看日志 |

关键权限点：

- 谁可以创建项目
- 谁可以上传版本
- 谁可以发布正式环境
- 谁可以回滚
- 谁可以删除版本
- 谁可以修改项目设置
- 谁可以管理成员

---

### 7.10 操作日志

系统必须记录：

```text
2026-07-01 14:09
杨东平 上传了版本 6JOWs-g

2026-07-01 14:20
张三 将版本 6JOWs-g 发布到 Production

2026-07-01 15:02
李四 回滚到版本 5Kp9a-x
```

日志应该可以按项目查看，也可以在系统后台全局查看。

## 9. 成功指标

第一阶段成功标准：

| 指标 | 目标 |
|---|---|
| 创建项目 | 可以正常创建和展示项目 |
| 上传产物 | 可以上传并生成独立版本 |
| 预览访问 | 每个版本有独立预览地址 |
| 正式发布 | 可以将某个版本发布为正式版 |
| 回滚 | 可以快速回滚到历史版本 |
| 日志 | 关键操作有记录 |
| SPA 支持 | React / Vue 深层路由刷新不 404 |
| 权限 | 至少区分管理员、开发者、只读成员 |
| 安全校验 | 能拦截危险文件和异常上传 |

最终判断标准：

> 它能否替代中小公司当前“人工上传服务器”的前端发布方式，并且让发布过程更安全、更清晰、更可回滚。

```text

React + Vite + TailwindCSS

Bun + Elysia

PostgreSQL

第一阶段：本地磁盘

```

---

## 11. 一句话总结

中小公司的企业级前端部署工具，不需要一开始做成 Vercel，但必须做到：

> 项目清晰、版本可靠、发布可控、回滚迅速、权限明确、日志完整。
